# Firestore Security Audit

## Codebase Scan

- Runtime: Next.js/TypeScript using Firebase Auth and modular Firestore Web SDK.
- Authentication: every Firestore listener and mutation requires a Firebase `user.uid`.
- Paths:
  - `users/{uid}/transactions/{id}`
  - `users/{uid}/budgets/{id}`
  - `users/{uid}/goals/{id}`
  - `users/{uid}/wishlistGroups/{groupId}`
  - `users/{uid}/wishlistGroups/{groupId}/items/{itemId}`
  - `users/{uid}/wishlist/{id}` (legacy compatibility)
- Queries:
  - transactions filtered by date window and ordered by `date desc`
  - budgets ordered by `category asc`
  - goals ordered by `deadline asc`
  - wishlist groups ordered by `title asc`
  - items in the active group ordered by `deadline asc`
- Mutations: create/read/delete on all primary collections; scoped updates for transactions, budgets, goals, wishlist groups, and wishlist items.
- No user profile document or public data is stored.
- Budget documents created by the current app include `month` in `YYYY-MM` format.

## Devil's Advocate Results

- Public list exploit: blocked by `isOwner(userId)`.
- Cross-user read/write/delete: blocked by UID path ownership.
- Update bypass: every update calls the full post-update validator, preserves immutable `createdAt`, and restricts changed fields.
- Ownership hijacking: strict allowed-field lists reject `uid` and ownership comes from the path.
- Immutable timestamp modification: goal and wishlist updates require unchanged `createdAt`.
- Type juggling: every field is type-checked.
- Required-field omission: rejected by `hasAll`.
- Resource exhaustion: all strings are bounded.
- Schema pollution: rejected by `hasOnly`.
- Invalid group state: title, description, color, deadline, and timestamp checks enforced.
- Invalid checklist state: priority enum, amount range, timestamp, and boolean checks enforced.
- Timestamp manipulation: `createdAt` must be recent; business dates must be Firestore timestamps.
- Negative/overflow amount: rejected by bounded amount validators.
- Goal progress overflow: `current` must remain between zero and `target`; target, title, deadline, and timestamp are immutable.
- Orphaned subcollection access: nested items require both UID ownership and an existing parent group.
- Query mismatch: each query targets the authenticated user's own subcollection and is compatible with the read rules.
- Checklist update escalation: only checklist content fields and `completed` may change.
- Goal update escalation: only goal content fields and `current` may change; `current <= target` remains enforced.
- Budget month integrity: new budgets require a bounded `YYYY-MM` month field.
